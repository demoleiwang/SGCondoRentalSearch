### New findings (round-2 specific)

- **(P1) False advertising in Smart Commute card.** The 🚇 card says "enter two locations (office + school) and find optimal areas between them", but the Streamlit UI has no two-location input. Clicking `▶ Bishan 2br 4000` just runs a normal single-location query — the two-location commute logic exists only in `commute.py` as a module and in the `/smart-search` slash command. A persona clicking that button will conclude the feature is broken. Either wire the two-location UI in, remove the card, or label it "CLI only".
- **(P2) "Condo + HDB Data" is not a feature.** That card describes the **data source**, not a user action. In a feature grid it reads as "we have data" which is table stakes. The data-volume reassurance belongs in the caption or a stats strip, not in a capability card.
- **(P2) "Rent Trends" card doesn't demonstrate trends.** The example `Queenstown 1b1b 3300` triggers a normal search that happens to show the trend panel (collapsed by default). A better example would open a trend view or deep-link to a project with rich history. As is, the card advertises a feature but the click doesn't showcase it.
- **(P2) No visual cue that `▶` buttons auto-search.** The caption says "click any example to try it", but a newcomer might think the button copies text to the search box, not auto-submits. Consider `▶ Try: <query>` or explicit `→ runs a live search` helper text under the caption.

## Lesson for next run

Adding a "what can I do" module is a high-leverage UX improvement — every new user benefits. But **a feature showcase must be honest with the UI paths it advertises.** Any card whose button lands on a generic results page (not a feature-specific view) should either demonstrate the feature concretely or be removed. The showcase should also lead the user toward features that actually solve their problem (my commute time, my lifestyle radius, my bargaining target) — it shouldn't merely catalog implementation modules.
